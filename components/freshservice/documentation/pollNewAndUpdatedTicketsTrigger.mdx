This trigger polls the [Freshservice API](https://api.freshservice.com/) for new and updated tickets on a scheduled basis.

#### How It Works

1. The trigger runs on the configured schedule (e.g., every 5 minutes)
2. It fetches tickets from Freshservice using the `GET /tickets` endpoint with an `updated_since` parameter set to the last poll time, paginating through results via Link headers
3. Records are categorized as **created** or **updated** based on their timestamps — tickets whose `created_at` exceeds the last poll time are classified as created; all other tickets returned by the query are classified as updated
4. The trigger updates its internal state after each poll

#### Configuration

Configure the following inputs:

- **Connection**: The Freshservice connection used to authenticate API requests
- **Show New Records**: When enabled, includes newly created tickets in the results. Defaults to `true`
- **Show Updated Records**: When enabled, includes modified tickets in the results. Defaults to `true`

#### Returned Data

The trigger returns an object with two arrays — `created` for new tickets and `updated` for modified tickets.

#### Notes

- The `updated_since` parameter is set to the start time of the previous poll to ensure no tickets are missed between executions
- Pagination is handled automatically using Link headers returned by the Freshservice API
- A ticket is classified as **created** if its `created_at` timestamp exceeds the last poll time. If `created_at` does not exceed the cutoff but `updated_at` does, the ticket is classified as **updated**
- Refer to the [Freshservice API documentation](https://api.freshservice.com/) for rate limit details and available ticket fields
