This trigger polls [Karbon](https://karbonhq.com/) on a configured schedule and emits records that were modified since the previous run. It is a convenient alternative to webhooks when a scheduled pull is preferred over an event push.

It supports the resource types whose list responses expose a modification timestamp:

- **Contacts**: compared on `LastModifiedDateTime`.
- **Invoices**: compared on `UpdatedAt`.

(Work Items are intentionally not supported: their list response exposes no modification timestamp, and Karbon offers no server-side modified-date filter for them.)

#### How It Works

Karbon's list endpoints do not support filtering by a modification date, so on each scheduled run the trigger fetches all records of the selected resource and keeps only those whose timestamp is after the previous poll time. The current run time is stored as the cursor for the next run.

- **First run**: the very first execution records the cursor and emits nothing, so an instance does not receive a backlog of historical records on deploy.
- **Subsequent runs**: only records modified since the last run are returned.

Karbon exposes only a modification timestamp and no creation timestamp. A newly created record carries a recent modification timestamp just like a record that was edited, so newly created and edited records cannot be distinguished. Both are surfaced together under `updated`.

#### Returned Data

The trigger returns the matching records under `data.updated`. Fields shown are representative. The full response object includes additional properties.

#### Notes

- Detection is based on the resource's modification timestamp. Because the list responses do not expose a creation timestamp, newly created and edited records cannot be separated, and all modified records arrive together under `data.updated`.
- Because there is no server-side date filter, each run lists all records of the resource and filters them in memory.
